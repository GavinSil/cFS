Error status constants (from psp/fsw/modules/sim_stepping/cfe_psp_sim_stepping.h)
- CFE_PSP_SIM_STEPPING_STATUS_TIMEOUT = -4
  * Returned when WaitStepComplete exceeds the specified timeout_ms
  * Observed in T10 timeout testing (see task-t10-uds-timeout.txt)

- CFE_PSP_SIM_STEPPING_STATUS_DUPLICATE_BEGIN = -2
  * Returned when BeginStepSession called while prior session unresolved
  * Used to prevent step reentry during active step execution

Timeout error path semantic anchor: T10 UDS timeout
- Evidence file: .sisyphus/evidence/task-t10-uds-timeout.txt
- Verified: Wait with insufficient timeout_ms returns status=-4 (TIMEOUT)
- UDS WAIT opcode with timeout_ms=1 returns status=-4 over socket
- Core diagnostic marker: CFE_PSP: SIM_STEPPING_DIAG class=timeout status=-4
- State after timeout: state=2 (RUNNING), trig=1 (trigger pending)
- This confirms timeout semantics propagate correctly through UDS adapter

UDS WAIT path reuses inproc implementation
- UDS WAIT handler in cfe_psp_sim_stepping.c forwards to CFE_PSP_SimStepping_InProc_WaitStepComplete()
- Same timeout logic used by both inproc and UDS adapters
- Timeout parameter passed through from UDS request (struct.pack('<BxxxI', opcode, timeout_ms))
